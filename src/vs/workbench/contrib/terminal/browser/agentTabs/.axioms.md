Directory-scoped invariants for the stokd-ide fork's agent-aware terminal selector

## AX-IDE-WEBVIEW-TERMINAL-SELECTOR: The Selector Hosts The code-ext Sessions Webview, With A Stock Fallback

The agent terminal selector hosts a configuration-designated webview view
(`terminal.integrated.agentTabs.viewId`, default `stokd.agentDashboard.terminalTabs`) in the
terminal tabs strip via `IWebviewViewService.resolve`, reusing the EXISTING code-ext Sessions panel
webview rather than re-drawing a native list. It MUST fall back to the stock `TerminalTabbedView` â€”
byte-identical to upstream â€” whenever the flag is off, no view id is designated, or no resolver is
registered for that view id (`IWebviewViewService.hasResolver`). A fork running without the Sessions
extension therefore renders the stock tabs, never an empty/broken strip.

### Why

The Sessions panel UI lives in code-ext (`stokd.agentDashboard`); the fork is a thin enabler and
must reuse that webview, not reimplement it (which is what the earlier native plain-text list did).
And a fork shipped to anyone without the extension must look exactly like upstream â€” a broken host
in the terminal panel is unacceptable.

### How to apply

- `AgentTerminalWebviewHost` resolves the designated id through the extension's provider and
  reports `isAvailable` (delegating to `hasResolver`); `attach()` is a no-op returning `undefined`
  when unavailable so the caller falls back.
- `shouldUseAgentTabs({ flagEnabled, designatedViewId, hasResolver })` (`agentTabsSeam.ts`) is the
  pure gate; the `terminalView.ts` seam creates `AgentTerminalTabbedView` only when it returns true,
  else the stock `TerminalTabbedView`.
- `WebviewViewService.hasResolver(viewType)` backs the gate. The extension registers its webview
  provider at activation (startup), before the terminal panel opens, so the resolver is present at
  creation in the normal case; a rare race (panel opened mid-activation) falls back to stock until a
  window reload.
- `AgentTerminalHostController` continues to host the live xterm; only the selector strip changes
  from a native list to the hosted webview.

## AX-IDE-SELECTOR-WIDTH-PERSISTS: The Selector Column Width Is Set Once And Persisted, Never Reset On Relayout

The agent-terminal selector column width is applied to the SplitView ONLY on the first
`layout()` (restoring the persisted value), and is preserved across every subsequent relayout
(panel/window resize, visibility toggle, terminal added). `AgentTerminalTabbedView.layout()` MUST
NOT call `resizeView(selectorIndex, ...)` unconditionally â€” with `proportionalLayout: false`,
`splitView.layout(width)` already preserves each view's size and feeds the delta to the
high-priority terminal. The user's chosen width is persisted on every sash drag
(`onDidSashChange` â†’ `IStorageService.store`, `StorageScope.PROFILE`/`StorageTarget.USER`) under
`stokd.agentTabs.selectorWidth`, and restored on the next first-layout.

### Why

`layout()` is called on every resize/visibility change. The original implementation re-ran
`resizeView(selector, min(DEFAULT, width/2))` on every call, so the width the user had dragged was
stomped back to the default "randomly" (on any relayout) â€” a deal-breaker for using the strip.

### How to apply

- The width logic lives in the pure, DOM-free `SelectorWidthController`
  (`agentTerminalSelectorWidth.ts`): `onLayout(total)` returns a clamped width on the FIRST call
  and `undefined` thereafter (preserve); `onSashChange(size)` persists. Both `clampSelectorWidth`
  (cap at `[MIN, min(MAX, floor(total/2))]`) and `resolveStoredSelectorWidth` are pure.
- `AgentTerminalTabbedView` constructs the controller with storage-backed read/write, subscribes
  `splitView.onDidSashChange` to `onSashChange(getViewSize(selectorIndex))`, and in `layout()`
  applies `onLayout(width)` only when it returns a value.
- Never reintroduce an unconditional `resizeView` in `layout()`.
