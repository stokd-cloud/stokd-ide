# AXIOMS

## AX-MOD-EXT-001: Only `copilot` is fork-owned; other extensions are read-only upstream
Every extension under `extensions/` except `copilot/` is inherited from `microsoft/vscode` and must not be modified outside a governed task, because edits widen the fork's rebase conflict surface (SC_OVERVIEW.md, SC_TEST.md §0).

### Applies To
- `extensions/*/` (all extension subdirectories except `extensions/copilot/`)

### Acceptance Checks
- manual: A diff touching any extension other than `extensions/copilot/**` must cite a governed task ID and a rebase-impact justification; otherwise reject.
- `git diff --name-only HEAD~1 -- extensions/ | grep -v '^extensions/copilot/' | grep -v '^extensions/\.axioms\.md$'` returns empty for routine fork work.

### Enforcement
computational

## AX-MOD-EXT-002: `copilot` builds and tests on its own pipeline, separate from gulp
The `copilot` extension is compiled and watched independently of the gulp extensions pipeline; changes to it must be validated through its own scripts, not assumed covered by `transpile-extensions`.

### Applies To
- `extensions/copilot/**`

### Acceptance Checks
- `npm run compile-copilot` (i.e. `npm --prefix extensions/copilot run compile`) exits 0.
- `npm run test-extension` (vscode-test) passes for the copilot suite before merge.

### Enforcement
suite

## AX-MOD-EXT-003: An extension's manifest is its public contract
`package.json` fields `main`, `contributes`, `activationEvents`, `enabledApiProposals`, and `extensionDependencies` are observable contracts; changing them can break activation, load order, or the `vscode` API agreement and requires re-validating activation and API-proposal availability.

### Applies To
- `extensions/*/package.json`

### Acceptance Checks
- manual: Every name in `enabledApiProposals` must resolve to a proposal `.d.ts` shipped by core (`src/vscode-dts/`); a mismatch fails activation.
- `npm run test-extension` (runs `vscode-api-tests`, the `vscode` API contract test) passes after any manifest change touching `contributes`/`enabledApiProposals`/`main`.

### Enforcement
suite

## AX-MOD-EXT-004: Shared build infra is global — changes must rebuild all extensions
The shared esbuild runners (`esbuild-*.mts`), `postinstall.mjs`, and `tsconfig.base.json` are consumed by every extension; modifying them requires a full extensions rebuild and must preserve the TypeScript files `postinstall.mjs` deliberately keeps (`typescript.js`/`.d.ts`, `lib.*.d.ts`).

### Applies To
- `extensions/esbuild-common.mts`
- `extensions/esbuild-extension-common.mts`
- `extensions/esbuild-webview-common.mts`
- `extensions/postinstall.mjs`
- `extensions/tsconfig.base.json`

### Acceptance Checks
- `npm run transpile-extensions` and `npm run compile-copilot` exit 0 after the change.
- manual: After editing `postinstall.mjs`, confirm `html-language-features` and `extension-editing` still resolve `typescript.js`/`typescript.d.ts` (the files the prune script intentionally retains).

### Enforcement
manual

## AX-MOD-EXT-005: The root `extensions/package.json` is a shared-deps manifest, not an aggregator or extension
The root `extensions/package.json` declares only the shared toolchain dependencies and the `postinstall` hook; it must never grow per-extension manifest fields (`main`, `contributes`, `activationEvents`, `enabledApiProposals`), because each extension owns its own manifest and the root is consumed only as the shared dependency/build surface.

### Applies To
- `extensions/package.json`

### Acceptance Checks
- `node -e "const p=require('./extensions/package.json'); process.exit((p.main||p.contributes||p.activationEvents||p.enabledApiProposals)?1:0)"` exits 0 (none of the extension-manifest fields are present at the root).
- `node -e "const p=require('./extensions/package.json'); process.exit(p.scripts && p.scripts.postinstall==='node ./postinstall.mjs'?0:1)"` exits 0 (the shared TS-prune postinstall hook is intact).

### Enforcement
computational

## AX-MOD-EXT-006: A new chat-session provider is added without touching the shared renderer
Each multi-provider LLM CLI provider under `extensions/copilot/src/extension/chatSessions/<provider>/` keeps its provider-specific logic (descriptor, model catalog, event normalizer, permission modes, steering) in a pure `common/` layer with Vitest `*.spec.ts` coverage and I/O quarantined in `node/`; providers register through `agentCliProviderRegistry.ts` and the single chat renderer is never modified to add a provider (`agentCliProvider.ts`: "the renderer is never modified when a new provider is added").

### Applies To
- `extensions/copilot/src/extension/chatSessions/{claude,copilotcli,gemini,grok}/**`
- `extensions/copilot/src/extension/chatSessions/common/agentCliProvider.ts`
- `extensions/copilot/src/extension/chatSessions/common/agentCliProviderRegistry.ts`

### Acceptance Checks
- `find extensions/copilot/src/extension/chatSessions/grok/common/test -name '*.spec.ts' | grep -q .` and `find extensions/copilot/src/extension/chatSessions/gemini/common/test -name '*.spec.ts' | grep -q .` (each provider ships pure `common/` Vitest specs).
- `test -f extensions/copilot/src/extension/chatSessions/common/agentCliProviderRegistry.ts` (the shared registration seam exists).
- manual: A new provider adds a `<provider>/common/` descriptor + normalizer and registers via the registry; it does not edit the shared renderer or duplicate per-provider DI blocks.

### Enforcement
computational

## AX-MOD-EXT-007: Provider gating and default launch surface follow the derived setting contract
A provider's enable-gate is derived as `chat.agentHost.${id}Agent.enabled` via `providerEnabledSettingId()` (never a hand-written per-provider constant), and the Agents Window launch surface defaults to `chat` for every provider via the revertible `chat.agentSessions.defaultSurface` (`'chat' | 'terminal'`, default `'chat'`); changing either formula/default is a coordinated change with the core `agentSessions` surface.

### Applies To
- `extensions/copilot/src/extension/chatSessions/common/agentCliProvider.ts`
- `extensions/copilot/src/extension/chatSessions/*/common/*ProviderDescriptor.ts`

### Acceptance Checks
- `grep -q "providerEnabledSettingId" extensions/copilot/src/extension/chatSessions/common/agentCliProvider.ts` (the derived-setting helper is present).
- `grep -q "chat.agentHost\." extensions/copilot/src/extension/chatSessions/common/agentCliProvider.ts` (the `chat.agentHost.${id}Agent.enabled` formula is intact).
- manual: The default-surface setting `chat.agentSessions.defaultSurface` (core `defaultLaunchSurface.ts`) still defaults to `'chat'` and remains revertible to `'terminal'`.

### Enforcement
computational

## AX-MOD-EXT-008: Each copilot-owned SQLite store is independent and versioned
The four fork-owned copilot databases (chronicle session store, OTel spans sink, workspace chunk + embedding cache, external ingest index) are physically distinct `node:sqlite` databases carrying a version row (`schema_version` / `CacheMeta.version` / `Metadata.cacheVersion`); schema changes must bump that version and migrate forward, with no cross-database foreign keys and no shared connection with the core `src/` databases.

### Applies To
- `extensions/copilot/src/platform/chronicle/node/sessionStore.ts`
- `extensions/copilot/src/platform/otel/node/sqlite/otelSqliteStore.ts`
- `extensions/copilot/src/platform/workspaceChunkSearch/node/workspaceChunkAndEmbeddingCache.ts`
- `extensions/copilot/src/platform/workspaceChunkSearch/node/codeSearch/externalIngestIndex.ts`

### Acceptance Checks
- `grep -q "schema_version" extensions/copilot/src/platform/chronicle/node/sessionStore.ts` and `grep -q "schema_version" extensions/copilot/src/platform/otel/node/sqlite/otelSqliteStore.ts` (versioned schemas present).
- manual: A schema change bumps the store's version row and migrates forward; no `REFERENCES` crosses a database boundary (the chronicle store's only FK target is its own `sessions` table). Re-generate `.stokd/meta/extensions/SC_SCHEMA.puml` when entities change.

### Enforcement
computational
