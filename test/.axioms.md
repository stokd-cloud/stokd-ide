# AXIOMS

## AX-MOD-TEST-001: Node unit runner glob and exclude contract is load-bearing
The node Mocha runner discovers tests only via `TEST_GLOB = '**/test/**/*.test.js'`, and its `excludeGlobs` (browser/electron layers, native-module tests, and the web-only `vs/sessions/test/web.test.js`) determine which tests run in which harness; tests not in a `test/` subfolder, or moved across a layer boundary, are silently skipped rather than failed.

### Applies To
- `unit/node/index.js`
- `unit/electron/index.js`
- `unit/browser/index.js`

### Acceptance Checks
- `grep -n "TEST_GLOB" test/unit/node/index.js` returns `'**/test/**/*.test.js'`
- `grep -n "vs/sessions/test/web.test.js" test/unit/node/index.js` (the fork-specific CSS-importing E2E exclusion is present)
- `npm run test-node -- --run src/vs/sessions/test/common/agentHostSessionsProvider.test.ts` discovers and runs the named file

## AX-MOD-TEST-002: The automation driver definition must stay in sync with the source driver contract
`automation` publishes the `vscode-automation` driver whose type definition is copied from `src/vs/workbench/services/driver/common/driver.ts`; the copy step must be re-run whenever that source contract changes, or downstream suites (`smoke`, `sanity`, `mcp`) compile against stale types.

### Applies To
- `automation/tools/copy-driver-definition.js`
- `automation/src/**`

### Acceptance Checks
- `cd test/automation && npm run compile` exits 0 (runs `copy-driver-definition` then `tsc`)
- manual: after editing `src/vs/workbench/services/driver/common/driver.ts`, re-run `npm run copy-driver-definition` in `test/automation` before relying on smoke/sanity/mcp

## AX-MOD-TEST-003: Higher suites depend on a freshly compiled automation package
`smoke`, `sanity`, and `mcp` import the compiled `automation` package (`main: ./out/index.js`); a change to `automation/src` requires recompiling `automation` (and re-installing each sub-package's own lockfiled deps) before those suites will pass.

### Applies To
- `smoke/**`
- `sanity/**`
- `mcp/**`

### Acceptance Checks
- `grep -n "cd ../automation && npm run compile" test/smoke/package.json test/mcp/package.json` (compile chains the automation build)
- manual: `cd test/automation && npm run compile` before running `npm run smoketest`

## AX-MOD-TEST-004: Fork Agents Window / Chat automation selectors are observable contracts
The fork-distinguishing page objects (`automation/src/agentsWindow.ts`, `automation/src/chat.ts`) and their smoke scenarios encode DOM selectors for `src/vs/sessions/` surfaces (e.g. `.agent-sessions-workbench`, `.sessions-chat-widget`, `.sessions-chat-send-button`); changing those selectors here without a matching `src/vs/sessions/` change (or vice-versa) breaks Agents Window / Chat smoke coverage.

### Applies To
- `automation/src/agentsWindow.ts`
- `automation/src/chat.ts`
- `smoke/src/areas/agentsWindow/**`
- `smoke/src/areas/chat/**`

### Acceptance Checks
- `grep -n "agent-sessions-workbench" test/automation/src/agentsWindow.ts` (selector still present)
- manual: run the Agents Window smoke scenario against a build and confirm `openCurrentFolderInAgentsWindow` / `switchToAgentsWindow` resolve

## AX-MOD-TEST-005: Copyright-header provenance distinguishes upstream from fork-authored test files
Test files derived from upstream keep the `Copyright (c) Microsoft Corporation` header; net-new fork-authored files use the stokd thin-patch-fork header (per `SC_TEST.md` §3). Header provenance must not be rewritten when editing, because it marks the rebase conflict surface.

### Applies To
- `automation/src/**`
- `smoke/src/**`
- `sanity/src/**`
- `mcp/src/**`

### Acceptance Checks
- `head -4 test/automation/src/agentsWindow.ts` shows a copyright header
- manual: a new fork-authored test file carries the stokd header, not the Microsoft header

## AX-MOD-TEST-006: CI-generated component-fixture screenshots are not hand-edited
`componentFixtures/blocks-ci-screenshots.md` is auto-generated by CI (it declares `auto-generated by CI — do not edit manually`); the visual-regression baselines and screenshot index must be regenerated through the component-explorer pipeline, never edited by hand.

### Applies To
- `componentFixtures/blocks-ci-screenshots.md`
- `componentFixtures/component-explorer.json`

### Acceptance Checks
- `head -1 test/componentFixtures/blocks-ci-screenshots.md` contains `auto-generated by CI`
- manual: regenerate via the `@vscode/component-explorer-cli` pipeline rather than editing the file
