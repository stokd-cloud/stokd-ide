# AXIOMS

## AX-MOD-REMOTE-001: `.npmrc` is the single source of truth for the server Node.js runtime
The `target`, `ms_build_id`, `runtime`, and `build_from_source` keys in `remote/.npmrc` are read
verbatim by the build (`getNodeVersion()` in `build/gulpfile.reh.ts`) to select, download, and
checksum the Node binary shipped inside the server. Changing `target` requires a matching entry in
`build/checksums/nodejs.txt` and a native-module rebuild against the new ABI; the keys must not be
renamed or removed.

### Applies To
- `remote/.npmrc`
- `remote/web/.npmrc`

### Acceptance Checks
- `grep -E '^(target|ms_build_id|runtime|build_from_source)=' remote/.npmrc` returns all four keys.
- `manual: if remote/.npmrc 'target' changed, confirm the new version exists in build/checksums/nodejs.txt and the REH build (build/gulpfile.reh.ts) packages without a checksum/version error.`

### Enforcement
computational

## AX-MOD-REMOTE-002: Package names and folder layout are a build contract
The build hard-codes `REMOTE_FOLDER = remote/` and `WEB_FOLDER = remote/web/` and globs
`remote/package.json` / `remote/web/package.json` by path. The package names `vscode-reh` and
`vscode-web`, and the `remote/` + `remote/web/` directory layout, must not change without a
coordinated update to `build/gulpfile.reh.ts` and `build/gulpfile.vscode.web.ts`.

### Applies To
- `remote/package.json`
- `remote/web/package.json`

### Acceptance Checks
- `grep '"name": "vscode-reh"' remote/package.json` succeeds.
- `grep '"name": "vscode-web"' remote/web/package.json` succeeds.
- `grep -q "remote', 'web'" build/gulpfile.vscode.web.ts && grep -q "REMOTE_FOLDER" build/gulpfile.reh.ts`

### Enforcement
computational

## AX-MOD-REMOTE-003: The `overrides` block gates native-module compilation
The `overrides` in `remote/package.json` (e.g. `node-gyp-build`, `kerberos`→`node-addon-api`,
`ssh2`→`cpu-features`, `yauzl`) pin transitive build-time dependencies so native modules compile
from source across all REH target platforms. Entries must not be removed or loosened without
verifying native builds still succeed on every target platform.

### Applies To
- `remote/package.json` (`overrides`)

### Acceptance Checks
- `node -e "process.exit(require('./remote/package.json').overrides ? 0 : 1)"` (overrides block present).
- `manual: after any overrides change, run the REH native-module build on win32/darwin/linux x64+arm64 and confirm all native deps compile.`

### Enforcement
computational

## AX-MOD-REMOTE-004: Dependency changes must keep the committed lockfile in sync
`remote/package.json` and `remote/web/package.json` each have a committed `package-lock.json` that
fixes the reproducible production closure bundled into the server/web builds. Any add/remove/bump of
a dependency must regenerate and commit the corresponding lockfile in the same change; a stale
lockfile yields a non-reproducible build.

### Applies To
- `remote/package.json` + `remote/package-lock.json`
- `remote/web/package.json` + `remote/web/package-lock.json`

### Acceptance Checks
- `test -f remote/package-lock.json && test -f remote/web/package-lock.json`
- `node -e "require('./remote/package-lock.json'); require('./remote/web/package-lock.json')"` (both lockfiles exist and parse).
- `manual: after editing dependencies, run the install and confirm the matching package-lock.json updated and is staged for commit.`

### Enforcement
computational

## AX-MOD-REMOTE-005: Shared frontend dependencies stay version-aligned across `vscode-reh` and `vscode-web`
Every dependency declared in **both** `remote/package.json` and `remote/web/package.json` (the
`@xterm/*` family, `@microsoft/1ds-*`, `@vscode/iconv-lite-umd`, `@vscode/tree-sitter-wasm`,
`@vscode/vscode-languagedetection`, `jschardet`, `katex`, `tas-client`, `vscode-oniguruma`,
`vscode-textmate`) must declare an **identical** version range in both manifests, so the same
frontend (terminal renderer, telemetry, tokenizer) behaves identically in the server and browser
builds. Bumping a shared dependency in one manifest without the other is a drift defect.

### Applies To
- `remote/package.json` (`dependencies`)
- `remote/web/package.json` (`dependencies`)

### Acceptance Checks
- `node -e "const r=require('./remote/package.json').dependencies,w=require('./remote/web/package.json').dependencies;const bad=Object.keys(w).filter(k=>k in r&&r[k]!==w[k]);process.exit(bad.length?(console.error('version drift:',bad),1):0)"`

### Enforcement
computational

## AX-MOD-REMOTE-006: `remote/` is a manifest-only module — no runtime source
`remote/` ships dependency manifests, lockfiles, npm config, and this axiom file only; it carries no
TypeScript/JavaScript source of its own. Server/web runtime code belongs in `src/vs/server/**` and
`src/vs/workbench/**`, not here. Committing a `.ts`/`.js`/`.mjs`/`.cjs` source file under `remote/`
(outside `node_modules/`) violates the module's design intent of a minimal, source-free build-input
closure.

### Applies To
- `remote/**` (excluding `remote/node_modules/**`)

### Acceptance Checks
- `test -z "$(find remote -name node_modules -prune -o -type f \( -name '*.ts' -o -name '*.js' -o -name '*.mjs' -o -name '*.cjs' \) -print)"` (no committed source files under `remote/`).
- `git ls-files remote/ | grep -v 'node_modules/' | grep -Eqv '\.(npmrc|json|md)$' && exit 1 || exit 0` (every tracked, non-`node_modules` file is a manifest/lockfile/npmrc/markdown).

### Enforcement
computational

## AX-MOD-REMOTE-007: A `.npmrc` `target` bump must ship matching node checksums
The Node `target` version in `remote/.npmrc` is the lookup key for `getNodeChecksum()` in
`build/gulpfile.reh.ts`: darwin/linux binaries are checksummed as
`node-v<target>-{darwin,linux}-{arm64,x64}.tar.gz` in `build/checksums/nodejs.txt`. Any change to
`target` must add those four version-stamped rows (and refresh the un-version-stamped
`win-arm64/node.exe` / `win-x64/node.exe` SHA-256s) in the same change, or the REH build fails to
verify the downloaded Node binary. The Windows entries are keyed by bare `win-<arch>/node.exe` and
resolved through the internal `ms_build_id` build via `product.nodejsRepository`, so they are not
version-stamped.

### Applies To
- `remote/.npmrc` (`target`)
- `build/checksums/nodejs.txt`

### Acceptance Checks
- `node -e "const fs=require('fs');const t=/^target=\"(.*)\"$/m.exec(fs.readFileSync('remote/.npmrc','utf8'))[1];const c=fs.readFileSync('build/checksums/nodejs.txt','utf8');const miss=['darwin-arm64','darwin-x64','linux-arm64','linux-x64'].map(p=>'node-v'+t+'-'+p+'.tar.gz').filter(n=>!c.includes(n));process.exit(miss.length?(console.error('missing node checksums:',miss),1):0)"` (the four nix `.tar.gz` checksum rows exist for the current `target`).
- `manual: when bumping target, also refresh the win-arm64/node.exe and win-x64/node.exe SHA-256s in build/checksums/nodejs.txt and rebuild native modules against the new ABI.`

### Enforcement
computational
