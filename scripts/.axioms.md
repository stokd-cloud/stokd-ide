# AXIOMS

## AX-MOD-SCRIPTS-001: Launchers are cross-platform trios kept in lockstep
Every developer entry point is provided as a matching set across platforms — a
`*.sh` (POSIX), a `*.bat` (Windows), and where present a shared `*.js`
orchestrator — and a behavioral change to one variant MUST be mirrored in its
siblings so all platforms launch identically.

### Applies To
- `scripts/code*.sh`, `scripts/code*.bat`, `scripts/code*.js`
- `scripts/test*.sh`, `scripts/test*.bat`
- `scripts/node-electron.sh`, `scripts/node-electron.bat`

### Acceptance Checks
- manual: For each launcher/test `scripts/<name>.sh` touched, confirm `scripts/<name>.bat` (and `scripts/<name>.js` if it exists) carries the equivalent flags, default ports, and env vars. (POSIX-only maintenance scripts — `sync-upstream`, `verify-seam`, `code-agent-host`, `code-sessions-web`, `generate-definitelytyped` — are intentionally exempt and have no `.bat`.)
- `for n in code code-cli code-server code-web node-electron test test-integration test-remote-integration test-web-integration test-documentation; do [ -f "scripts/$n.sh" ] && [ ! -f "scripts/$n.bat" ] && echo "missing bat: $n"; done; true`

## AX-MOD-SCRIPTS-002: Server launch handshakes are an observable contract
Launchers detect readiness/address by matching a specific server stdout line —
`Web UI available at (.*)` for `code-server.js` and `READY:(\d+)` for
`code-agent-host.js`; these strings and their regexes, plus the default ports
(server `9888`, web `8080`, agent-host/sessions-web `8081`), MUST NOT change
without a coordinated update on both the server and launcher sides.

### Applies To
- `scripts/code-server.js`
- `scripts/code-agent-host.js`
- `scripts/code-sessions-web.js`

### Acceptance Checks
- `grep -q 'Web UI available at' scripts/code-server.js`
- `grep -q 'READY:' scripts/code-agent-host.js`
- manual: If a handshake string or default port changes, update the corresponding server entry point in `out/`/`src/` and any CI step that waits on it in the same task.

## AX-MOD-SCRIPTS-003: Fork-maintenance scripts stay non-destructive and seam-true
`sync-upstream.sh` MUST NOT push or rewrite git history (it only prints the
commands and may `--ff-only` fast-forward), and `verify-seam.sh` MUST keep
asserting the terminal agent-tabs seam invariant (`AX-TERMINAL-AGENT-TABS`):
flag `terminal.integrated.agentTabs.enabled` with `default: false`, the
flag-off path still `createInstance(TerminalTabbedView)`, and `terminalView.ts`
referencing the `ITerminalTabsView` interface.

### Applies To
- `scripts/sync-upstream.sh`
- `scripts/verify-seam.sh`

### Acceptance Checks
- `bash scripts/verify-seam.sh`
- `grep -q 'merge --ff-only' scripts/sync-upstream.sh`
- manual: Confirm `sync-upstream.sh` only *prints* `git push` / rebase commands for the user to run (inside `cat`/`echo` blocks) and never executes a push or a history rewrite itself; the sole git mutation it performs is the `--ff-only` fast-forward under `--ff-main`.

## AX-MOD-SCRIPTS-004: Vendored agent-host protocol is generated, never hand-edited
Files under `src/vs/platform/agentHost/common/state/protocol/` are produced by
`sync-agent-host-protocol.ts` and carry a `DO NOT EDIT -- auto-generated` banner;
changes to the protocol MUST be made by re-running the sync script against the
sibling `../agent-host-protocol` repo, and the script's transform pipeline
(tab conversion, duplicate-import merge, tsfmt formatting, copyright + banner,
`.ahp-version` stamp) MUST be preserved.

### Applies To
- `scripts/sync-agent-host-protocol.ts`
- `src/vs/platform/agentHost/common/state/protocol/**`

### Acceptance Checks
- `grep -q 'DO NOT EDIT' scripts/sync-agent-host-protocol.ts`
- manual: Verify generated protocol files retain the `DO NOT EDIT -- auto-generated by scripts/sync-agent-host-protocol.ts` banner and were not edited by hand.
- manual: After a protocol change, run `npx tsx scripts/sync-agent-host-protocol.ts` and confirm `.ahp-version` matches the source repo's short SHA.

## AX-MOD-SCRIPTS-005: Perf/leak gates are config-driven and CI-consumed
The chat-simulation harness reads its thresholds and baseline from
`scripts/chat-simulation/config.jsonc`; baseline/threshold changes are
observable regression-gate behavior and MUST be made in that file (not
hard-coded), keeping `merge-ci-summary.js` able to consume per-group results.

### Applies To
- `scripts/chat-simulation/config.jsonc`
- `scripts/chat-simulation/test-chat-perf-regression.js`
- `scripts/chat-simulation/test-chat-mem-leaks.js`
- `scripts/chat-simulation/merge-ci-summary.js`

### Acceptance Checks
- `node -e "require('./scripts/chat-simulation/common/utils.js').loadConfig('perfRegression')"`
- manual: Confirm new perf/leak thresholds live in `config.jsonc` and the CI report job still merges via `merge-ci-summary.js`.
